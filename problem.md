## 문제점 정리
1. ai모델들을 서버 처음 시작시에 올렸지만 답변시간 30s~60s
-> 주요 원인 : 청크수 >> 보험 상품 추천(recommendation) 텍스트 2500, 상품 설명(product_info) 청크수 150개, 특약조정(special_terms) 텍스트 1000
2. 특약조정 기능 없음
-> 해결책 : db설계에 특약란을 만들어서 데이터 추가 저장
3. langchain의 기술적 적용x 
-> 해결책 : messages들이 쌓이면서 sql문을 생성하고 db를 조정하는 걸 적용해야할듯 
4. pdf내의 선택특약들을 전부 뽑고 그것들의 체크리스트를 보여줘서 db에 저장해야함 = ai가 보험 상품의 내용을 설명하면 그 안에서 특약을 선택하게 만들어야함.
-> 이어지는 의문 : 그럼 특정 source내의 선택특약들을 rag로 어떻게 뽑아낼건데? 
--> 해결책 : 메타 데이터 검색 > 검증 프로세스가 필요함 
--> 해결책2 : 특약 세션을 rag내에서 추출해낼 수 있다면? 어차피 보험상품 목차는 어느정도 일정한 부분이 존재할거 아니야.<<**“구조 인식형 RAG”**>>
--> 해결책3 : ✔ “계층형 chunking”
                    문서
                ├── Section (특약)
                │    ├── Subsection (암특약)
                │    ├── Subsection (입원특약)

                → 처음부터 이렇게 나누면

                retrieval = section 단위 가능
                정확도 크게 올라감
--> 해결책4 : multi-level embedding + section_id개념 + 각 문서별 목차로 메타데이터 태깅

1. chunk 단위로 먼저 검색 (넓게)
2. 상위 결과에서 section_id 추출
3. 해당 section 전체 확장
4. 필요하면 subsection 단위 refine

<고민> -> bge-m3모델은 질의를 임베딩하여 이것이 질의와 의미가 비슷하다고 결론을 냄. 근데 그것이 특약이라는 정확한 섹션을 구분하지못함. 그럼 임베딩을 섹션단위로 하면되지않나?
<얻은것> rag의 장점은 의미적 유연함임. 빡빡하게할거면 그냥 DB쓰는게 나음.
<얻은것> RAG와 LLM이 사용자의 의도를 구분하는 것의 차이는 LLM은 해석하고 RAG는 데이터를 기반으로 답을 찾음.
